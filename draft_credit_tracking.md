## 2. Remaining work

### 2.1 Session-start budget enforcement

`CreditForecastService` isn't consumed anywhere yet. Wire into `TradingSessionService.open(...)`:

```
const projection = await creditForecast.projectForSession({
  packId,
  expectedCandidatesPerHour,
  expectedHours,
  mode,
});
if (projection.birdeye.projectedMtd > projection.birdeye.monthlyBudget
 || projection.helius.projectedMtd > projection.helius.monthlyBudget) {
  if (!input.allowOverBudget) {
    throw new CreditBudgetExceeded({ projection });
  }
  await operatorEvents.emit({ severity: 'warning', detail: 'over-budget session opened' });
}
```

`allowOverBudget` is an explicit operator confirmation in the Session page UI.

Budgets: sourced from `compose.env` (`BIRDEYE_MONTHLY_BUDGET`, `HELIUS_MONTHLY_BUDGET`). Default to reasonable values matching the current plan tier.

### 2.2 Purpose tagging audit

Every paid call logs `purpose` on its `ProviderCreditLog` row. Grep for calls that log with `purpose = UNKNOWN` or omit purpose:

- `HeliusClient.*` — ensure callers always pass purpose (`ENRICHMENT_CREATOR_LINEAGE`, `EXEC_PRIORITY_FEE`, etc.).
- `BirdeyeClient.*` — same.

Missing purpose → fix at the call site. Don't add a default; untagged calls become invisible in forecasts.

### 2.3 Alert rules (provisioned)

Ship under `grafana/provisioning/alerting/credit-burn.yaml` along with the Credit Burn dashboard:

| Rule | Condition | Severity | Pending |
|---|---|---|---|
| `credit_burn_80pct` | today credits / daily budget > 0.8 | warning | 5 m |
| `credit_burn_100pct` | > 1.0 | critical | 2 m |
| `credit_slope_3x` | 1 h slope > 3× trailing-24h median | warning | 15 m |
| `provider_credit_error_rate` | `ProviderCreditLog.ok = false` share > 0.1 for 15 m | warning | 15 m |

Each rule annotates: `summary`, `description`, `runbook_url`, `__dashboardUid__`, `__panelId__`. All route to the single ops channel.

### 2.4 Daily budget derivation

Monthly budget ÷ days in month = daily budget. Surface on the Credit Burn dashboard header. Hardcode nothing — read from env at render time via the backend `/api/operator/budgets` endpoint (new).

### 2.5 Rollup job

After 30 days, raw `ProviderCreditLog` rows churn into `ProviderCreditDaily` (new table in [draft_database_plan.md §6](draft_database_plan.md)). Daily cron at 04:00 UTC runs:

```
INSERT INTO "ProviderCreditDaily"
SELECT date_trunc('day', createdAt) as day, providerName, purpose, SUM(credits), COUNT(*)
FROM "ProviderCreditLog" WHERE createdAt < now() - interval '30 days'
GROUP BY 1,2,3;

DELETE FROM "ProviderCreditLog" WHERE createdAt < now() - interval '30 days';
```

Run from `trading_bot/backend/scripts/prisma-maintenance.mjs` (invoked by n8n).

### 2.6 Per-pack credit attribution

`v_api_session_cost` exists. Add a new view `v_api_pack_cost_daily` that groups `ProviderCreditLog` by `strategyPackId + createdAt::date`. Requires that every call attach the active `strategyPackId` — today that attribution may be missing on background enrichment calls.

Wire: `ProviderBudgetService.requestSlot` accepts an optional `{ strategyPackId, tradingSessionId }` context; every caller passes it where known.
