## 5. Runtime Performance

#### PERF-C1 · Rate limiter busy-wait loop
**File**: `src/utils/rate-limiter.ts:14-29`

The `while(true)` pattern re-evaluates the window on every iteration, spinning at least twice per throttled call. Under sustained 30 RPM Helius load this adds visible event-loop lag to the trade execution path.

```typescript
// Replace the entire while(true) loop with a single-pass implementation:
async waitForSlot(): Promise<void> {
  const now = Date.now();
  this.timestamps = this.timestamps.filter(t => now - t < this.windowMs);

  if (this.timestamps.length < this.maxRequests) {
    this.timestamps.push(now);
    return;
  }

  // Sleep until the oldest timestamp expires, then re-enter once
  const waitMs = this.windowMs - (now - this.timestamps[0]) + 10;
  log.debug({ limiter: this.name, waitMs }, "rate limit — waiting");
  await new Promise(r => setTimeout(r, waitMs));
  await this.waitForSlot();  // tail-recurse once after sleep
}
```

#### PERF-M2 · `HeliusService.lastSlotByAddress` uses FIFO eviction
**File**: `src/services/helius.ts:173-191`

The 1000-entry cap evicts by insertion order (FIFO), not by recency of access. Frequently-accessed wallets can be evicted while stale ones remain.

**Fix**: Replace with an LRU map or use a `Map` with delete-and-reinsert on access (JS Maps maintain insertion order):

```typescript
private updateSlot(address: string, slot: number): void {
  this.lastSlotByAddress.delete(address);  // remove existing entry
  this.lastSlotByAddress.set(address, slot);  // reinsert at end (most recent)
  if (this.lastSlotByAddress.size > 1000) {
    // first entry is now the least recently used
    this.lastSlotByAddress.delete(this.lastSlotByAddress.keys().next().value!);
  }
}
```
